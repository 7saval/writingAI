## Slide 1: 문제와 해결책

### Before (기존 방식의 문제)
```
사용자 경험:
┌──────────────────────────────────────────┐
│ 제출 → [바운스 로딩 3-5초] → [타이핑 시뮬 15초]  │
│      (아무것도 못 봄)    (가짜 UI)        │
│                                          │
│ 총 소요 시간: 20초                      │
│ TTFF: 3-5초                            │
│ 문제: 타이핑 시뮬레이션 (실제 AI 과정 아님)  │
└──────────────────────────────────────────┘
```

**핵심 문제점:**
- 첫 반응까지 3-5초 대기 (현대 웹 기준 너무 김)
- 실제 생성 5초 + 가짜 타이핑 15초 = 총 20초
- "AI가 글을 써 내려가는" 몰입감 없음

---

### After (SSE 기반 실시간 스트리밍)
```
사용자 경험:
┌───────────────────────────────────────────┐
│ 제출 → [200ms] → 첫 글자 → [실제 생성 3-5초]   │
│      (즉각 반응)        (실시간 토큰 표시)  │
│                                          │
│ 총 소요 시간: 3-5초                      │
│ TTFF: 200ms                             │
│ 개선: 실제 AI 생성 과정 투명성 확보        │
└───────────────────────────────────────────┘
```

**개선 결과:**
- TTFF 95% 단축 (3-5초 → 200ms)
- 총 UX 시간 75% 단축 (20초 → 3-5초)
- 실시간 토큰 누적으로 진정한 몰입감 제공

## Slide 2: 기술 선택 & 구현

### 왜 SSE인가?

| 기술 | 단방향 | 자동재연결 | POST body | 복잡도 | 선택 |
|------|--------|----------|----------|--------|------|
| WebSocket | ✗ | - | ✓ | 높음 | ✗ |
| EventSource | ✓ | ✓ | ✗ | 낮음 | ✗ |
| **SSE** | ✓ | ✓ | ✓ | **낮음** | **✓** |

**선택 근거:**
- 단방향 스트리밍만 필요 (WebSocket 과도)
- fetch + ReadableStream으로 POST body 지원 가능
- HTTP 기반이라 프로토콜 업그레이드 불필요

---

### 구현 (7개 파일, ~450줄)

**백엔드 (3개 파일)**
- `aiService.ts`: OpenAI `stream: true` 옵션 + 각 토큰 yield
- `writingController.ts`: SSE 프로토콜 헤더 설정 + 4가지 이벤트 (user/ai_start/chunk/done)
- `writingRoutes.ts`: `POST /:id/write/stream` 라우트 추가

**프론트엔드 (4개 파일)**
- `types/database.ts`: `Paragraph.isStreaming?: boolean` 필드 추가
- `writing.api.ts`: `fetch + ReadableStream`으로 SSE 파싱 + 5개 콜백 (onUserParagraph/onAiStart/onChunk/onDone/onError)
- `Editor.tsx`: `handleSubmit` 콜백 기반 상태 관리로 완전 재구성
- `ParagraphItem.tsx`: `isStreaming` 분기 추가 + 실시간 content 표시

## Slide 3: 성과 & 배운 점

### 정량적 성과

|--------|--------|-------|--------|

| **바운스 로딩** | 있음 | 없음 | **제거** |

---

### 개발 중 마주친 버그 & 해결

**#1: CORS 정책 위반**
- 문제: `Access-Control-Allow-Origin: *` + `credentials: 'include'` 충돌
- 원인: 백엔드 직접 설정 + 프론트 쿠키 포함
- 해결: 백엔드 직접 설정 제거 (cors 미들웨어 충분), 프론트 credentials 제거 (JWT 인증)

**#2: AI 콘텐츠 미표시**
- 문제: SSE 수신되지만 화면에 커서만 깜박임
- 원인: `onAiStart`에서 ID 변경 → `onChunk`가 임시 ID로 찾을 수 없음
- 해결: `onAiStart`에서 ID 명시적 유지 (`{ ...p, ...paragraph, id: tempAiId }`)

---

### 기술적 배운 점

1. **CORS + credentials 정책**
   - 와일드카드와 credentials는 양립 불가
   - 미들웨어 수준 설정이 충분할 때 컨트롤러 재설정 피할 것

2. **콜백 기반 상태 관리 패턴**
   - 여러 콜백이 같은 객체 수정 시 추적 ID 명시적 유지 필수
   - 향후 프로젝트 재사용 가능한 패턴

3. **단방향 프로토콜의 강력함**
   - "양방향이 정말 필요한가?" 먼저 질문하기
   - SSE는 단순하면서도 강력한 도구

## 면접 예상 질문

### Q1. 왜 WebSocket 대신 SSE를 선택했나요?
**포인트:** 요구사항 → 기술 선택
- WritingAI는 서버 → 클라이언트 단방향이면 충분
- WebSocket은 양방향 지원으로 복잡도 높음
- SSE는 HTTP 기반이라 간단하면서 필요한 모든 기능 제공

### Q2. 콘텐츠 미표시 버그를 어떻게 해결했나요?
**포인트:** 문제 분석 → 해결책 → 일반화
- `onAiStart`에서 서버 데이터로 교체할 때 ID까지 바꿈
- `onChunk`에서 임시 ID로 찾을 수 없었음
- 해결: ID를 명시적으로 유지 (콜백 기반 상태 관리에서 중요한 패턴)

### Q3. 이 개선이 실제로 얼마나 유의미했나요?
**포인트:** 수치 + 사용자 영향
- 정량: TTFF 95% 단축, 총 시간 75% 단축
- 정성: 실시간 AI 작성 과정 투명성 + 몰입감 향상
- 사용자 입장에서 20초 대기 → 5초로 경험 완전히 달라짐
