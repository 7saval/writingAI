#### 🔧 해결한 문제

**AI 기능 고도화: 대화 문맥 구조화 및 생성 품질 개선**

- **문제**: AI가 앞선 맥락이나 설정집(Lorebook) 내용을 완벽히 반영하지 못하거나, 생성 결과의 일관성이 부족한 경우가 발생함. 또한 테스트 코드 작성 중 TypeScript 산술 연산 오류로 빌드가 실패하는 등의 사소한 이슈가 있었음.
- **해결**:
  1. **문맥 구조의 정교화**: AI에게 전달되는 메시지를 `System`, `User`, `Assistant` 역할별로 명확히 분리함. 특히 시놉시스와 설정집 정보는 첫 번째 `User` 메시지의 헤더로 배치하고, 이전 대화들은 유저의 입력과 AI의 답변 순서에 맞춰 `User/Assistant`로 매핑하여 AI가 대화의 흐름을 더 정확히 파악하도록 개선함.
  2. **사용자 지시사항(AI Directive) 도입**: 사용자가 "더 어두운 분위기로", "액션 씬 위주로"와 같은 실시간 피드백을 주면 이를 프롬프트 마지막에 결합하여 생성 결과에 반영되도록 구현함.
  3. **BASE_PROMPT 도입**: 모든 단락 생성 시 "개연성 있는 사건/행동 포함" 및 "한국어 준수" 지침을 강제하도록 개선했습니다.
  4. **assistant 역할 도입**: AI가 이전에 작성한 내용을 자신의 답변으로 정확히 인지하도록 메시지 구조를 리팩토링했습니다.
  5. **문맥 확장**: 참조하는 최근 단락 수를 늘려 대화의 흐름이 더 자연스럽게 유지되도록 했습니다.
  6. **테스트 코드를 통한 로직 검증**: 육안으로 확인하기 어려운 복잡한 프롬프트 구조와 Role 매핑을 `aiTest.ts`를 통해 자동화된 방식으로 검증했습니다.

**프론트엔드 성능 최적화: `useTypingAnimation` 리팩토링 및 실시간 동기화**

- **문제발생**: `useTypingAnimation` 훅에서 3개의 서로 다른 `useEffect`가 연쇄적으로 실행되면서 불필요한 리렌더링이 자주 발생함. 특히 단락 수정 후 새로운 내용이 즉시 UI에 반영되지 않고 새로고침 시에만 갱신되는 데이터 동기화 이슈가 있었음.
- **진행**: React 공식 문서에서 권장하는 "렌더링 단계에서 상태 업데이트(Render-phase state update)" 패턴을 적용함. 프로프 변경을 `useEffect`로 감시하는 대신 렌더링 로직 내에서 이전 값과 비교해 즉시 상태를 갱신하도록 구조를 변경하고, 타이핑 애니메이션을 위한 `useEffect` 1개만 남기고 나머지는 제거함.
- **성과**:
  1. **리렌더링 횟수 감소**: 불필요한 이펙트 실행 및 상태 업데이트 단계를 줄여 렌더링 성능을 개선함.
  2. **사용자 경험(UX) 향상**: 저장 버튼 클릭 시 수정된 내용이 지연 없이 즉각적으로 반영되어 훨씬 부드러운 인터페이스를 제공함.
  3. **코드 유지보수성**: 훅의 로직이 단순해지고 의존성 관리가 명확해짐.

**AI 기능 고도화: 글의 템포 조절을 위한 시놉시스 5단계 구조화 및 인텔리전트 컨텍스트 연동**

- **문제발생**: AI가 전체 시놉시스를 한꺼번에 참고하다 보니, 이야기의 초반부임에도 불구하고 결말 복선을 너무 빨리 회수하거나 갑자기 최종 결전으로 건너뛰는 등 **'글의 템포 조절'**이 되지 않는 고질적인 문제가 발생함. 작가는 현재 '발단'을 쓰고 있는데 AI는 '절정'의 내용을 가져와 쓰는 식의 맥락 붕괴가 잦았음.
- **진행**:
  1.  **시놉시스 데이터 구조 개편**: 단순 단일 텍스트였던 시놉시스를 소설의 5단 구성(발단, 전개, 위기, 절정, 결말)에 맞춰 저장하도록 `SynopsisModal`과 데이터 스키마를 개편함. 데이터를 JSON 문자열로 관리하여 각 단계의 독립성을 확보함.
  2.  **현재 집필 단계(Stage) 시스템 도입**: 유저가 `StoryContextPanel`에서 현재 집필 중인 단계를 직접 선택하게 함. 이 정보는 전역 상태(`useWritingStore`)로 관리되며 AI 생성 요청 시 백엔드로 전달됨.
  3.  **백엔드 컨텍스트 필터링 로직 (aiService.ts)**: AI에게 모든 시놉시스를 주지 않고, 현재 선택된 단계의 내용을 강조하는 인텔리전트 가이드를 삽입함.
  ```typescript
  // aiService.ts 로직 예시
  const stageKey = options.stage ? stageMap[options.stage] : null;
  if (stageKey && synopsisObj[stageKey]) {
    // 전체 개요는 보여주되, 현재 단계의 시놉시스를 [현재 집중 단계]로 별도 강조
    synopsisText = `[전체 개요]\n${Object.values(synopsisObj).join("\n")}\n\n[현재 집중 단계: ${options.stage}]\n${synopsisObj[stageKey]}`;
    stageGuide = `\n현재 이야기는 소설의 5단 구성 중 [${options.stage}] 단계에 해당합니다. 해당 단계의 시놉시스 내용을 중점적으로 반영하여 이야기를 전개해 주세요.`;
  }
  ```
- **성과**:
  1.  **글의 완급 조절(Pacing Control) 성공**: AI가 현재 단계에 명시된 사건에만 집중하게 함으로써 이야기가 너무 빨리 진행되거나 산으로 가는 현상을 방지함.
  2.  **맥락 유지력 향상**: 사용자가 "지금은 발단 부분이야"라고 명시적으로 알려주는 효과가 있어, AI가 불필요한 결말 스포일러를 생성하지 않고 현재 장면에 최적화된 묘사를 수행함.
  3.  **데이터 호환성 확보**: 기존 텍스트 시놉시스 사용자들을 위해 JSON 파싱 실패 시 전체 내용을 '발단' 섹션으로 자동 마이그레이션하는 예외 처리를 적용하여 사용자 불편을 최소화함.

**인증 보안: 새로고침(Hard Navigation) 및 수동 URL 변경 시 로그인 세션 유지 버그 수정**

- **문제발생**: 로그인 후 브라우저 주소창에서 URL을 직접 수정하여 이동하거나 페이지를 새로고침하면, Zustand 스토어의 `accessToken`이 소실되면서 로그인이 풀리고 "로그인이 필요한 서비스입니다" 알림창이 뜨는 현상이 발생함.
- **진행**:
  1.  **원인 분석**: 새로고침 시 메모리 상태가 초기화되는데, 이때 실행되는 `/verify-user` 요청이 토큰 없이 전송됨. 백엔드는 에러 로그 방지를 위해 `401 Unauthorized` 대신 `200 OK (authenticated: false)`를 반환하고 있었고, 이로 인해 Axios의 401 인터셉터(자동 갱신 로직)가 작동하지 않음.
  2.  **해결 시도 및 재분석**: 1차적으로 훅에서 `refresh`를 호출하도록 수정했으나, 재시도 요청 전 스토어를 업데이트하지 않아 헤더에 토큰이 누락되는 문제와 `verifyUser` 응답에 토큰이 없어 상태가 `undefined`로 덮어씌워지는 결함을 추가 발견함.
  3.  **최종 해결**: `useAuthUserQuery`의 오케스트레이션 로직을 고도화함. `refresh` 성공 즉시 스토어를 업데이트하여 다음 API 요청의 헤더를 확보하고, `storeLogin` 호출 시 `refresh`로 얻은 토큰을 명시적으로 유지하도록 개선함.

  1.  **사용자 경험(UX) 개선**: 어떤 방식의 페이지 진입(SPA 이동, 새로고침, URL 직접 입력)에서도 끊김 없는 인증 상태를 유지함.
  2.  **인증 로직의 견고함 확보**: 토큰 만료나 소실 상황에서도 리프레시 토큰이 유효하다면 자동으로 세션을 복구하는 안정적인 흐름을 구축함.

#### 💡 개념 정리

**1. LLM 대화 모델의 Role 구조 (OpenAI API 기준)**

- **System**: AI의 페르소나, 답변 언어, 준수해야 할 규칙 등 전역적인 지침을 설정합니다.
- **User**: 사용자의 질문이나 지시사항입니다.
- **Assistant**: AI의 이전 답변 내용입니다.
- **중요성**: 대화 맥락(Context)을 전달할 때 단순히 텍스트를 나열하는 것이 아니라, 이 Role들을 정확히 구분해 주어야 AI가 "누가 한 말인지"를 파악하고 훨씬 더 일관성 있는 답변을 생성할 수 있습니다.

**2. AI 기능 개발 시 '테스트 코드'의 가치와 장점**

- **문맥 구조의 가시화**: 수백 줄에 달하는 프롬프트가 실제로 어떤 순서와 형태로 AI에게 전달되는지 콘솔 출력을 통해 한눈에 파악할 수 있습니다.
- **예외 케이스 조기 발견**: 설정집이 비어있거나, 단락이 없을 때 등 다양한 상황을 모의(Mock) 데이터로 시뮬레이션하여 런타임 에러를 방지합니다.

**3. 렌더링 단계에서 상태 업데이트 (Render-phase state update)**

- **개념**: `useEffect`는 브라우저가 화면을 그린 후에 실행되지만, 렌더링 중에 특정 상태를 다른 데이터(Props 등)와 동기화해야 할 때는 렌더링 함수 내에서 직접 상태를 업데이트하는 것이 더 효율적일 수 있습니다.

**❌ 일반적인 방식 (useEffect 사용)**

```tsx
function ParagraphItem({ content }) {
  const [text, setText] = useState(content);

  // 화면이 한 번 그려진 후에야 실행됨 (한 박자 늦음)
  useEffect(() => {
    setText(content);
  }, [content]);

  return <div>{text}</div>;
}
```

- **흐름**: Props 변경 → **이전 state로 렌더링** → 브라우저 그리기 → Effect 실행 → **다시 렌더링** → 브라우저 다시 그리기
- **문제**: 데이터가 바뀔 때 아주 잠깐 동안 이전 데이터가 보였다가 바뀌는 '반짝임(Flicker)'이 발생할 수 있습니다.

**✅ 개선된 방식 (렌더링 단계에서 업데이트)**

```tsx
function ParagraphItem({ content }) {
  const [text, setText] = useState(content);
  const [prevContent, setPrevContent] = useState(content);

  // 렌더링 도중에 직접 비교하고 즉시 업데이트
  if (content !== prevContent) {
    setPrevContent(content);
    setText(content); // React는 이 시점에서 렌더링을 중단하고 새로운 state로 즉시 다시 시작함
  }

  return <div>{text}</div>;
}
```

- **흐름**: Props 변경 → 렌더링 도중 `if`문 통과 → `setState` 호출 → **브라우저에 그리기 전에 즉시 새 state로 재렌더링** → 최종 결과만 브라우저에 그리기
- **성과**: 렌더링 사이클을 획기적으로 줄이고, 사용자에게 항상 최신 데이터만 보여줄 수 있어 성능과 UX가 모두 개선됩니다.

**사용자 경험(UX) 극대화를 위한 가변형 레이아웃 및 반응형 사이드바 구현**

- **문제**: 기존 `WritingSession` 페이지는 3컬럼 고정 레이아웃으로, 해상도가 낮아지면 에디터 영역이 좁아져 집필이 불가능하거나 모바일 환경에서 레이아웃이 완전히 깨지는 문제가 있었음. 또한, 데스크탑에서도 하드코딩된 수치(`1024px`)를 사용하여 유지보수성이 떨어지는 결함이 있었음.
- **해결**:
  1. **상태 기반 조건부 레이아웃**: `isLeftOpen`, `isRightOpen` 상태와 `useMediaQuery` 훅을 결합하여 환경에 적합한 상태를 선언적으로 관리함.
  2. **복합적 트랜지션 처리**:
     - **데스크탑**: `lg:relative`와 `width` 변화를 활용하여 에디터 영역이 실시간으로 확장되는 유동적인 레이아웃 구축.
     - **모바일**: `fixed`와 `translate-x`를 활용하여 사이드바가 화면 위를 덮는 Drawer(Overlay) 형태로 구현함.
  3. **코드 품질 개선**: 하드코딩된 매직 넘버를 `BREAKPOINTS` 상수로 중앙 집중화하고, 이벤트 리스너 로직을 `useMediaQuery` 훅으로 캡슐화하여 재사용성을 높임.

- **작성 코드 요약 (`WritingSession.tsx`)**:

const isMobile = useMediaQuery(`(max-width: ${BREAKPOINTS.LG - 1}px)`);

// 2. 레이아웃 엔진 핵심 로직

  className={`fixed inset-y-0 left-0 z-40 transition-all duration-300 ease-in-out lg:relative overflow-hidden ${
    isLeftOpen
      ? "w-64 translate-x-0 opacity-100"
      : "w-0 -translate-x-full lg:translate-x-0 opacity-0 lg:opacity-100"
  }`}

      onClick={toggleLeft}

#### 💡 개념 정리

**1. 선언적 미디어 쿼리 관리 (useMediaQuery Custom Hook)**
하드코딩된 브레이크포인트 수치(`1024px` 등)는 나중에 디자인 시스템이 변경될 때 모든 파일을 찾아다니며 수정해야 하는 리스크가 있습니다. `window.matchMedia` API를 캡슐화한 `useMediaQuery` 훅과 중앙 집중형 `BREAKPOINTS` 상수를 사용하면, UI 로직을 명령형(`addEventListener`)이 아닌 선언적 방식으로 작성할 수 있어 유지보수성이 비약적으로 향상됩니다.

**2. window.matchMedia API**
- **역할**: CSS 미디어 쿼리가 현재 브라우저 상태와 일치하는지 여부를 자바스크립트에서 판단하고 감시할 수 있게 해주는 API입니다.
- **성능적 이점**: 브라우저 창 크기가 1px이라도 변할 때마다 호출되는 `window.onresize`와 달리, 브라우저 엔진이 최적화된 방식으로 쿼리 상태가 '바뀌는 순간'에만 이벤트를 발생시킵니다. 따라서 불필요한 연산을 줄여 성능을 최적화할 수 있습니다.
- **활용**: 이를 React 상태 시스템과 결합하면, 특정 해상도 진입 여부에 따라 UI를 선언적으로 변경할 수 있는 '반응형 인터랙션 시스템'을 구축할 수 있습니다.

**3. 레이아웃 시프트와 Overflow 제어**
사이드바를 접을 때 `width: 0`을 적용하면 내부 텍스트나 버튼들이 영역 밖으로 삐져나오는 현상이 발생합니다. 이때 부모 요소에 `overflow: hidden`을 적용하고 내부 컨텐츠는 고정 너비를 가진 Wrapper로 감싸주면, 내부 레이아웃을 파괴하지 않으면서도 자연스럽게 가려지는 애니메이션을 안정적으로 구현할 수 있습니다.

**4. 사용자 가시성(Visibility)과 Z-index 전략**
강력한 인터랙션을 제공하는 토글 버튼은 레이아웃 변화의 주체보다 항상 더 높은 레이어에 위치해야 합니다. 특히 사이드바가 화면 위를 덮는 모바일 환경에서는 버튼의 `z-index`를 최상위 레벨(`z-50`)로 관리하여, 어떤 상황에서도 사용자가 컨트롤을 잃지 않도록 설계하는 것이 인터랙티브 웹 디자인의 핵심입니다.
